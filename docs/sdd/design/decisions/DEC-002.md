# DEC-002: ディスクI/Oサンプリング方式

## ステータス

**ステータス**: 承認済
**決定日**: 2026-03-23

## コンテキスト

ディスクI/O使用率の取得には `/proc/diskstats` の2回サンプリングが必要。サンプリング間隔がフック全体の実行時間に直接影響する。

### 現状の課題
- `/proc/diskstats` は累積値のため、瞬時値を得るには2回読み取りが必要
- サンプリング間隔が短いと精度が低下し、長いと実行時間要件に抵触する
- NFR-PERF-001は100ms以内を要求

### 制約条件
- ディスクI/OチェックはBash実行時のみ（Agent起動時にはチェックしない）
- 精度よりも実行速度を優先する

## 検討した選択肢

### 選択肢A: 100msサンプリング

**概要**: `sleep 0.1` で100ms間隔のサンプリング

**メリット**:
- 実装がシンプル
- 一定の精度を確保

**デメリット**:
- sleep自体で100ms消費し、前後の処理を含めるとNFR-PERF-001の100msを超過する可能性が高い

---

### 選択肢B: 50msサンプリング

**概要**: `sleep 0.05` で50ms間隔のサンプリング

**メリット**:
- 前後の処理を含めても100ms以内に収まる可能性が高い
- Bash実行前の軽い警告として十分な精度

**デメリット**:
- 50msのサンプリングでは精度がやや低い（ただし警告用途には十分）

---

### 選択肢C: 前回値のキャッシュ

**概要**: `/tmp/ccresmon_diskstats` にprocfsの値をキャッシュし、前回の呼び出し時の値との差分で算出。sleepを不要にする。

**メリット**:
- sleepが不要で実行時間を大幅に短縮
- 呼び出し間隔が自然なサンプリング間隔になる

**デメリット**:
- 初回呼び出し時はキャッシュがないためチェックスキップが必要
- /tmp への書き込みが必要（副作用）
- キャッシュの破損・不整合への対処が必要

---

## 決定

**選択**: 選択肢C（前回値のキャッシュ）

**根拠**:
1. sleepを排除することで、NFR-PERF-001の100ms制限を確実に達成できる
2. Claude Code使用中はフックが頻繁に呼ばれるため、キャッシュの更新頻度は十分
3. 初回はチェックスキップで安全にフォールバック（フェイルオープン原則）
4. `/tmp` は揮発性であり、再起動時に自動クリーンアップされる

### キャッシュファイル設計

```
ファイルパス: /tmp/ccresmon_diskstats_<UID>
内容: <timestamp_ms> <io_ticks>
例:   1711180800000 12345678
```

- UIDを含めることでマルチユーザー環境での衝突を回避
- 不正なフォーマットの場合はキャッシュを無視（フェイルオープン）

## 影響

### 影響を受けるコンポーネント
- [resource-collector](../components/resource-collector.md) @../components/resource-collector.md: get_diskio_usage()の実装を変更

### リスク
- キャッシュファイル破損時にI/Oチェックがスキップされる → 許容範囲（フェイルオープン）
- 長時間Claude Codeを使用しなかった場合、古いキャッシュとの差分が不正確になる → タイムスタンプで古いキャッシュを検出し破棄（60秒以上前のキャッシュは無効とする）
