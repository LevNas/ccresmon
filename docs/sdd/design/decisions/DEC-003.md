# DEC-003: フェイルオープン設計の採用

## ステータス

**ステータス**: 承認済
**決定日**: 2026-03-23

## コンテキスト

ccresmonはClaude Codeの動作前に実行されるフック。スクリプト自体の不具合やリソース情報取得の失敗時にどう振る舞うかを決定する必要がある。

### 現状の課題
- procfsの読み取り失敗、不正なJSON入力など、予期しないエラーが発生する可能性がある
- フックがエラーで動作不能になった場合、Claude Code全体の動作を止めるべきではない

### 制約条件
- ccresmonは「安全装置」であり、安全装置の故障でメインの機能を止めてはならない
- 安全性重視の設計方針

## 検討した選択肢

### 選択肢A: フェイルオープン（エラー時は許可）

**概要**: エラー発生時はすべてのツール実行を許可する。

**メリット**:
- Claude Codeの動作を妨げない
- ccresmonの不具合がユーザー体験を損なわない
- シンプルで予測可能な動作

**デメリット**:
- エラー時にリソース監視が機能しない

---

### 選択肢B: フェイルクローズ（エラー時はブロック）

**概要**: エラー発生時はすべてのツール実行をブロックする。

**メリット**:
- 安全側に倒れる

**デメリット**:
- ccresmonの不具合でClaude Codeが使用不能になる
- ユーザーがフックを無効化するまで作業が完全にブロックされる

---

## 決定

**選択**: 選択肢A（フェイルオープン）

**根拠**:
1. ccresmonは補助的な安全装置であり、それ自体の不具合で生産性を損なうべきではない
2. リソース監視が一時的に機能しなくても、PCがフリーズする前にユーザー自身が異変に気づける
3. エラー発生は通常一時的（procfsの瞬間的な不可用など）であり、次回の呼び出しで回復する可能性が高い

### 実装パターン

```bash
# スクリプト全体をtrapで囲む
trap 'exit 0' ERR

# 各関数でもエラー時は安全なデフォルト値を返す
get_memory_usage() {
  local result
  result=$(...) 2>/dev/null || { echo 0; return; }
  echo "$result"
}
```
